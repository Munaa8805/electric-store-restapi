Conventions

Most successful responses use { "success": true, ... }. Errors use { "success": false, "message": "..." } (and sometimes HTTP 4xx/5xx).

Protected routes

Send header: Authorization: Bearer <token> (JWT from POST /auth/login or POST /auth/register).

Content-Type

Use Content-Type: application/json for bodies.

Authentication

POST /api/v1/auth/register

Create an account. Returns a JWT.

Request body

{
  "name": "Jane Doe",
  "email": "jane@example.com",
  "password": "min-length-per-server"
}

Password length is enforced in authController (currently a minimum of 4 characters — adjust in code for production).

Response 201

{
  "success": true,
  "data": {
    "user": {
      "id": "…",
      "name": "Jane Doe",
      "email": "jane@example.com",
      "role": "user",
      "phone": null,
      "address": null,
      "city": null,
      "state": null,
      "zip": null,
      "country": null,
      "image": null,
      "isActive": true,
      "isVerified": false,
      "createdAt": "…",
      "updatedAt": "…"
    },
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9…"
  }
}

Errors

400 missing fields / short password · 409 email already registered

POST /api/v1/auth/login

Sign in. Rate limited (same as register).

Request body

{
  "email": "jane@example.com",
  "password": "your-password"
}

Response 200

{
  "success": true,
  "data": {
    "user": { "id": "…", "name": "…", "email": "…", "role": "user", "…": "…" },
    "token": "eyJhbGciOiJIUzI1NiIs…"
  }
}

Errors

401 invalid credentials · 403 account disabled (isActive: false)

GET /api/v1/auth/logout

Acknowledges logout; clear the token on the client. Clears auth cookies if your app sets them.

Response 200

{ "success": true, "message": "Logged out successfully" }

Products

List routes such as /featured and /discounted should be defined before /:id in the router so they are not parsed as IDs.

GET /api/v1/products

Paginated product list.

Query

Response 200

  "meta": {
    "total": 120,

    "pages": 12,

Orders

All routes require Authorization: Bearer <token>.

POST /api/v1/orders

Create an order for the authenticated user.

Request body

{
  "items": [
    {
      "product": "<productObjectId>",
      "name": "Product name",
      "quantity": 2,
      "image": "https://example.com/image.jpg",
      "price": 99.99
    }
  ],
  "shippingAddress": {
    "address": "123 Main St",
    "city": "Ulaanbaatar",
    "postalCode": "14200",
    "country": "MN"
  },
  "paymentMethod": "card",
  "paymentResult": {
    "id": "payment-ref",
    "status": "completed",
    "update_time": "…",
    "email_address": "payer@example.com"
  },
  "itemsPrice": 199.98,
  "taxPrice": 16,
  "shippingPrice": 9.99,
  "totalPrice": 225.97
}

Response 201

{ "success": true, "data": { "_id": "…", "user": "…", "orderItems": [ … ], "…": "…" } }

Validation

paymentMethod must be card, cash, or bank_transfer. taxPrice and totalPrice are required numbers.

GET /api/v1/orders/my-orders

Orders for the current user.

Response 200

{ "success": true, "data": [ { "_id": "…", "orderItems": [ … ], "…": "…" } ] }

GET /api/v1/orders/:id

Single order (user populated with name, email in controller).

PUT /api/v1/orders/:id/pay

Marks order paid (isPaid, paidAt).

PUT /api/v1/orders/:id/deliver

Marks order delivered.

GET /api/v1/orders

All orders (admin-style listing in controller). Ensure this route is registered before GET /:id in Express or / may never match as intended.
